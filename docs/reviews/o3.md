## 1 Foundational Pillars (all documents agree)

| Pillar                         | Consensus Across Docs                                                                                                                                                                                                        |
| ------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Hierarchical machines**      | `Server ⟶ Signer ⟶ Entity ⟶ Account` – nested FSMs, each with its own block/log.                                                                                                                                             |
| **Single-proposer per Entity** | Always signer #0 of the quorum; no round-robin in the MVP.                                                                                                                                                                   |
| **Quorum definition**          | List `[(signer,weight)…] + threshold]`, stored as **hash** in `EntityProvider.sol`; full struct supplied with every signature set.                                                                                           |
| **Consensus flow**             | Proposer dry-runs block → broadcasts → each validator re-runs, signs if state-hash matches → when ≥ threshold sigs, block finalised.                                                                                         |
| **Storage & persistence**      | Everything mutated in-memory → every 100 ms server tick:<br>  • snapshot to LevelDB (`entity_state/`)<br>  • append `ServerBlock` to history (`server_blocks/`)<br>  • write WAL of incoming `ServerTx[ ]` before execution. |
| **Hash of truth**              | After tick, fold *all* `[signer ⇒ entity ⇒ state]` into deterministic RLP → SHA-256 → `serverHash` (test/diagnostic only).                                                                                                   |
| **Networking in the MVP**      | Simulated: Outbox messages loop straight back into Server inbox; no real libp2p yet.                                                                                                                                         |
| **Jurisdiction layer**         | Only two on-chain contracts for now: `EntityProvider.sol` (quorum registry) & `Depositary.sol` (reserve, disputes).                                                                                                          |

## 2 Key Data-Flow (re-verified)

```
User → ServerTx                           (level-0)
ServerTx → EntityInput (routed by signer) (level-1)
EntityInput.kind = add_tx | propose_block | commit_block
add_tx.tx = EntityTx                      (level-2)
```

* `ServerBlock` = `[EntityInput,…] + height + timestamp`
* `EntityBlock` = `[EntityTx,…] + hash` – stored in `entity_blocks/{id}/`

After the server finishes its tick:

1. New `ServerBlock` is persisted.
2. `OutboxMessage[]` (generated during Entity execution) is serialised to new `ServerTx[]` and re-inserted into `Server.mempool`.

No empty blocks are produced; if nothing arrived the tick simply snapshots.

### Quick-start Mental Checklist for Any New Developer

1. **One proposer, many validators** per Entity.
2. **Three queues**: Server.mempool → Entity.mempool → Account deltas.
3. **Everything in memory**, LevelDB only for snapshots/WAL.
4. **Every tick**: apply inputs → persist `ServerBlock` → compute `serverHash` → flush Outbox.
5. **No empty blocks, no rounds, no fancy networking** in MVP.
6. **Disputes & reserves** handled only by `Depositary.sol`; nothing else touches L1.
7. **Hash = truth**; identical code + identical inputs ⇒ identical hashes ⇒ ready to sign.

Stick to these seven and you’re on the rails.
